# 任务看板

## 状态定义

- `BLOCKED`：依赖尚未满足。
- `READY`：可以领取。
- `IN_PROGRESS`：已有终端负责。
- `REVIEW`：已提交，等待主终端验证。
- `DONE`：已合并且验证通过。

任务状态只有在真实进展发生后更新。负责人填写终端编号或分支名。

## 启动规则

- 终端 0 先完成 T0-01 至 T0-04，并创建基线提交。
- 终端 1～5 从同一个基线提交创建 Git worktree，不能共用工作目录。
- 每个终端启动后先阅读 `README.md`、`02-architecture.md`、`03-contracts.md`、`04-development-plan.md` 和本看板。
- 每次只领取一个 `READY` 任务；开始后填写负责人并改为 `IN_PROGRESS`。
- 需要修改 contracts 时只提交提案，不直接在功能分支形成另一套公共类型。
- 一个任务只有在代码、测试、交接信息都完整后才能进入 `REVIEW`。

## 终端任务包

### 终端 0：架构、契约和集成

- 首批任务：T0-01～T0-04。
- 持续职责：契约提案裁决、分支合并、全量验证、任务状态和知识库维护。
- 交付包：基线提交、contracts API、Worker 协议、每日集成结果、AI 日志汇总。
- 关键风险：过早修改公共接口导致所有分支返工。

### 终端 1：画布与编辑体验

- 首批任务：T1-01、T1-02；之后执行 T1-03、T1-04。
- 交付包：画布组件、节点/边交互、命令系统、撤销重做、画布组件测试。
- 依赖输入：PipelineNode/PipelineEdge 契约和节点注册表。
- 关键风险：把 React Flow 内部类型写入项目文件或共享契约。

### 终端 2：执行引擎

- 首批任务：T2-01、T2-02；之后执行 T2-03～T2-08。
- 交付包：图校验、执行计划、节点执行器、缓存策略和完整单元测试。
- 依赖输入：contracts 和 Worker 消息协议。
- 关键风险：依赖 DOM/React、执行用户代码、错误处理不确定或结果不可复现。

### 终端 3：文件与项目持久化

- 首批任务：T3-01、T3-02；之后执行 T3-03～T3-05。
- 交付包：CSV/JSON/XLSX 适配器、导出安全、项目文件打开保存和夹具。
- 依赖输入：DataSchema、数据表适配接口和项目格式版本。
- 关键风险：把原始数据写入项目文件、错误编码、Excel 日期和公式注入。

### 终端 4：节点配置与数据预览

- 首批任务：T4-01、T4-02；之后执行 T4-03～T4-05。
- 交付包：配置表单框架、运行面板、虚拟化预览、Join 配置和组件测试。
- 依赖输入：节点配置 Schema、诊断 DTO 和执行进度消息。
- 关键风险：复制全量数据到 React state、通过解析错误文案驱动 UI。

### 终端 5：质量、性能和交付

- 首批任务：T5-01、T5-02；之后执行 T5-03～T5-06。
- 交付包：Playwright、性能生成器、安全场景、浏览器矩阵、演示脚本和测试报告。
- 依赖输入：固定测试 ID、测试夹具和完整业务链路。
- 关键风险：为了让测试通过修改产品语义，或者记录未经测量的性能数字。

## Definition of Done

任务标记为 `DONE` 必须同时满足：

- 行为符合产品范围和核心契约。
- 没有越权修改其他终端拥有的模块。
- 新增行为有对应自动化测试，错误路径至少覆盖一个。
- lint、typecheck 和相关测试真实通过。
- 不含密钥、真实业务数据、调试输出和无关生成文件。
- 交接信息包含提交、文件、测试结果、风险和 AI/人工分工。
- 主终端已合并并在主分支复验。
