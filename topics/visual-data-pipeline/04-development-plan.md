# 开发计划

## 总体安排

以 6 个 Codex 终端并行、12 个工作日完成 MVP 为基准。时间是计划值，不作为已经实现的效率数据。

| 阶段 | 时间 | 目标 | 退出条件 |
|---|---:|---|---|
| 0. 契约冻结 | 第 1 天 | 初始化仓库和共享契约 | 所有模块可编译引用 contracts |
| 1. 技术骨架 | 第 2～3 天 | 画布、Worker、文件适配器和测试框架跑通 | CSV 输入到预览的最小链路通过 |
| 2. 核心节点 | 第 4～6 天 | 完成主要转换节点和配置 UI | 单输入演示流水线完整运行 |
| 3. 高级能力 | 第 7～8 天 | Join、XLSX、缓存、撤销重做 | 双输入演示和项目保存通过 |
| 4. 集成加固 | 第 9～10 天 | 性能、安全、错误恢复和 E2E | 验收测试通过，无阻断缺陷 |
| 5. 交付准备 | 第 11～12 天 | 演示、文档和 AI 使用总结 | 全新环境构建并完成演示彩排 |

## 阶段 0：契约冻结

- 初始化 pnpm workspace、TypeScript、lint、format 和 test。
- 建立 `apps/web`、`packages/contracts`、`pipeline-engine`、`file-adapters`。
- 实现项目文件、节点注册、诊断和 Worker 消息 Schema。
- 确定示例数据和验收场景。
- 建立 CI 基础命令。

这一步完成前，其他终端只做设计审查或独立原型，不能各自定义共享类型。

## 阶段 1：最小垂直链路

必须先跑通：

```text
选择 CSV → Worker 解析 → 生成字段 Schema → 表格预览 → 显示统计
```

同时完成：

- 画布基本操作。
- Worker 通信和取消协议。
- CSV 解析错误展示。
- 测试夹具和首个 Playwright 冒烟测试。

## 阶段 2：核心流水线

优先顺序：

1. 选择/重命名、类型转换。
2. 筛选、计算字段。
3. 排序、去重。
4. 分组统计。
5. CSV 和 JSON 导出。

每增加一个节点，必须同时完成配置 Schema、执行器、配置 UI 和测试，不留下“只有 UI”或“只有算法”的半成品。

## 阶段 3：高级能力

- Join 双输入端口和字段冲突策略。
- XLSX 导入导出。
- 上游变化导致下游缓存失效。
- 项目文件导入导出和格式版本校验。
- 撤销重做、未保存提示和异常恢复。

## 阶段 4：集成加固

- 使用 10 万行固定数据进行性能测试。
- 检查主线程长任务和内存峰值。
- 测试取消后迟到 Worker 消息。
- 测试损坏项目文件、错误编码、混合类型和空文件。
- 检查表达式注入和 Spreadsheet 公式注入。
- 完成完整演示场景 E2E。

## 阶段 5：交付

- 固化演示输入文件和期望输出。
- 生成测试和性能结果，不手写虚假数字。
- 完成隐私说明、使用说明和架构图。
- 汇总 AI 的实际贡献、错误建议、人工修正与验证证据。
- 在干净环境执行安装、构建、测试和演示。

## 每日协作节奏

1. 主终端发布当日基线和允许修改的契约版本。
2. 各终端领取 `05-task-board.md` 中一个可执行任务。
3. 开始前同步主分支，确认文件所有权。
4. 完成后运行本模块测试并提交，不直接修改其他终端分支。
5. 提交交接信息：变更、测试、风险、契约建议和 AI 日志条目。
6. 主终端依赖顺序合并，并运行全量验证。

## 分支和 Worktree

建议在 `project/` 首次基线提交后创建：

```bash
git worktree add ../project-canvas -b feature/canvas
git worktree add ../project-engine -b feature/engine
git worktree add ../project-files -b feature/file-adapters
git worktree add ../project-nodes -b feature/node-config
git worktree add ../project-qa -b feature/qa-delivery
```

主终端保留在 `project/`，负责契约、集成和最终验收。禁止多个终端在同一个工作目录并行写文件。
