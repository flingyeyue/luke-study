# 开发计划

## 总体安排

以 6 个 Codex 终端并行、12 个工作日完成 MVP 为基准。时间是计划值，不作为已经实现的效率数据。

当前状态：M0 工程基线和 Wave 1 并行骨架已于 2026-07-11 完成，M1 最小垂直链路进行中。Wave 1 已分别建立画布、引擎、文件适配器、配置与预览面板和浏览器测试骨架，但产品界面尚未把 CSV 导入、Worker 执行和真实预览串成一条链路，因此不将 M1 标记为完成。

| 阶段        |         时间 | 目标                                   | 退出条件                     |
| ----------- | -----------: | -------------------------------------- | ---------------------------- |
| 0. 契约冻结 |      第 1 天 | 初始化仓库和共享契约                   | 所有模块可编译引用 contracts |
| 1. 技术骨架 |   第 2～3 天 | 画布、Worker、文件适配器和测试框架跑通 | CSV 输入到预览的最小链路通过 |
| 2. 核心节点 |   第 4～6 天 | 完成主要转换节点和配置 UI              | 单输入演示流水线完整运行     |
| 3. 高级能力 |   第 7～8 天 | Join、XLSX、缓存、撤销重做             | 双输入演示和项目保存通过     |
| 4. 集成加固 |  第 9～10 天 | 性能、安全、错误恢复和 E2E             | 验收测试通过，无阻断缺陷     |
| 5. 交付准备 | 第 11～12 天 | 演示、文档和 AI 使用总结               | 全新环境构建并完成演示彩排   |

## 里程碑与交付物

### M0：工程基线，第 1 天结束

状态：`DONE`（2026-07-11）

交付物：

- 可安装、可构建、可测试的 pnpm workspace。
- `contracts` 初版 Schema 和契约测试。
- Worker 请求、进度、结果和取消消息协议。
- 固定测试夹具和知识库入口。

质量门禁：`pnpm lint`、`pnpm typecheck`、`pnpm test` 和 `pnpm build` 均成功。M0 未完成前不创建并行功能分支。

### M1：最小垂直链路，第 3 天结束

状态：`IN_PROGRESS`（Wave 1 骨架已完成）

交付物：CSV 输入节点、Worker 解析、字段推断、1000 行预览、运行状态和首个 E2E。

质量门禁：真实浏览器中导入固定 CSV 后可看到 Schema、行数和预览；解析期间界面可操作；失败能定位到输入节点。

### M2：可用单输入流水线，第 6 天结束

交付物：选择/重命名、类型转换、筛选、计算、排序、去重、聚合、CSV/JSON 导出及全部配置 UI。

质量门禁：每个节点同时具备配置 Schema、执行器、UI、成功测试和错误测试；演示链路 E2E 通过。

### M3：功能完整 MVP，第 8 天结束

交付物：Join、XLSX、缓存失效、项目保存/打开、撤销重做和未保存保护。

质量门禁：双输入订单演示可重复执行；项目文件不包含原始数据；重新打开后图和配置一致。

### M4：候选发布版，第 10 天结束

交付物：性能报告、安全测试、异常恢复和全量 E2E。

质量门禁：无阻断缺陷；10 万行测试有真实设备和浏览器数据；表达式与公式注入测试通过。

### M5：考察交付版，第 12 天结束

交付物：生产构建、演示数据、演示脚本、架构说明、测试结果和 AI 使用总结。

质量门禁：在干净环境从安装到完整演示可重复执行，所有已知限制有记录。

## 阶段 0：契约冻结

- 初始化 pnpm workspace、TypeScript、lint、format 和 test。
- 建立 `apps/web`、`packages/contracts`、`pipeline-engine`、`file-adapters`。
- 实现项目文件、节点注册、诊断和 Worker 消息 Schema。
- 确定示例数据和验收场景。
- 建立 CI 基础命令。

这一步完成前，其他终端只做设计审查或独立原型，不能各自定义共享类型。

## 阶段 1：最小垂直链路

必须先跑通：

```text
选择 CSV → Worker 解析 → 生成字段 Schema → 表格预览 → 显示统计
```

同时完成：

- 画布基本操作。
- Worker 通信和取消协议。
- CSV 解析错误展示。
- 测试夹具和首个 Playwright 冒烟测试。

## 阶段 2：核心流水线

优先顺序：

1. 选择/重命名、类型转换。
2. 筛选、计算字段。
3. 排序、去重。
4. 分组统计。
5. CSV 和 JSON 导出。

每增加一个节点，必须同时完成配置 Schema、执行器、配置 UI 和测试，不留下“只有 UI”或“只有算法”的半成品。

## 阶段 3：高级能力

- Join 双输入端口和字段冲突策略。
- XLSX 导入导出。
- 上游变化导致下游缓存失效。
- 项目文件导入导出和格式版本校验。
- 撤销重做、未保存提示和异常恢复。

## 阶段 4：集成加固

- 使用 10 万行固定数据进行性能测试。
- 检查主线程长任务和内存峰值。
- 测试取消后迟到 Worker 消息。
- 测试损坏项目文件、错误编码、混合类型和空文件。
- 检查表达式注入和 Spreadsheet 公式注入。
- 完成完整演示场景 E2E。

## 阶段 5：交付

- 固化演示输入文件和期望输出。
- 生成测试和性能结果，不手写虚假数字。
- 完成隐私说明、使用说明和架构图。
- 汇总 AI 的实际贡献、错误建议、人工修正与验证证据。
- 在干净环境执行安装、构建、测试和演示。

## 每日协作节奏

1. 主终端发布当日基线和允许修改的契约版本。
2. 各终端领取 `05-task-board.md` 中一个可执行任务。
3. 开始前同步主分支，确认文件所有权。
4. 完成后运行本模块测试并提交，不直接修改其他终端分支。
5. 提交交接信息：变更、测试、风险、契约建议和 AI 日志条目。
6. 主终端依赖顺序合并，并运行全量验证。

## 合并顺序

同一 Wave 按以下依赖顺序合并，避免在主分支形成不可运行状态：

1. contracts 和测试夹具。
2. pipeline-engine。
3. file-adapters。
4. Worker 客户端和运行时状态。
5. 画布与节点配置 UI。
6. E2E、性能和交付文档。

每次合并后执行受影响包测试；每天结束前执行一次全量 lint、typecheck、unit 和 build。E2E 从 M1 开始保持可运行，不在最后一天集中补写。

## 风险与缓冲

| 风险                            | 最早验证时间 | 失败时的降级方案                                  |
| ------------------------------- | -----------: | ------------------------------------------------- |
| 10 万行数据导致浏览器内存压力   |   第 2～3 天 | 限制并发中间结果，降低预览上限，优先保留 CSV 链路 |
| XLSX 包体积或日期语义不满足要求 |      第 3 天 | XLSX 降为导入 Beta，MVP 保证 CSV/JSON             |
| 数据运算库 Worker 兼容性差      |      第 2 天 | 使用项目节点接口实现 MVP 必要算法                 |
| Join 语义和内存开销过大         |      第 6 天 | MVP 只支持 inner/left 和单键或有限多键            |
| 双向虚拟化交互不稳定            |      第 3 天 | 固定行高，先保证纵向虚拟化和列数上限              |
| 多终端修改共享契约产生冲突      |         全程 | contracts 只由主终端合并，其他终端提交变更提案    |

计划预留第 11 天作为集成缓冲。若 M3 延迟，按顺序降级 XLSX 高级选项、缓存优化和非关键快捷键，不削减数据正确性、安全检查与核心 E2E。

## 分支和 Worktree

建议在 `project/` 首次基线提交后创建：

```bash
git worktree add ../project-canvas -b feature/canvas
git worktree add ../project-engine -b feature/engine
git worktree add ../project-files -b feature/file-adapters
git worktree add ../project-nodes -b feature/node-config
git worktree add ../project-qa -b feature/qa-delivery
```

主终端保留在 `project/`，负责契约、集成和最终验收。禁止多个终端在同一个工作目录并行写文件。
